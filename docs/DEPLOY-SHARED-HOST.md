# Deploying ClipForge on a shared host (Coolify / Traefik / existing stack)

This is the deploy pattern for a box that **already runs other services** — e.g.
the `yaatal-vps` Contabo box running Coolify + Yaatal + Paperclip + a
Claude Code (CC) operator + the Hermes agent, all reachable via Telegram over
Tailscale. It differs from the vanilla 3-command `deploy.sh` flow.

## Why not `scripts/deploy.sh` here

`deploy.sh` is written for a **dedicated** box. On a shared host it would:
- bring up a **bundled nginx on :80/:443** — collides with the host's existing
  reverse proxy (Coolify's `coolify-proxy` owns 80/443/8080), and
- run **`ufw --force enable`** opening 80/443/8766 — can disrupt the existing
  services and Tailscale.

So on a shared host we deploy manually with `docker-compose.prod.yml` (no nginx,
private API bind) and skip the firewall/SSL/systemd steps that assume a
dedicated box.

## Deploy

```bash
cd /opt/clipforge
git pull
# secrets already generated by scripts/bootstrap.sh (config/.env)
docker compose --env-file config/.env -f docker-compose.prod.yml up -d --build
```

- API is published on `127.0.0.1:3000` only (set `API_BIND=<tailnet-ip>` in
  `config/.env` first if you also want to reach it from your laptop over Tailscale).
- Postgres/Redis are Docker-internal — no host ports, no clash with the host's
  own Postgres/Redis.
- `init.sql` runs automatically on first Postgres init.

Verify:
```bash
docker exec clipforge-api curl -sf http://localhost:3000/health
docker exec -e SMOKE_CLEANUP=1 clipforge-api node scripts/smoke-check.js   # self-cleaning
```

## Posting / DMs — Hermes handles it

**ClipForge does not drive the phones on this box.** The host runs the
`hermes-agent` (Python, exposing device tools to CC over **MCP-stdio**), not the
WebSocket relay that ClipForge's `src/integration/hermes.js` expects. Therefore:

- Leave `HERMES_RELAY_URL` **unset** → ClipForge's post path stays in **dry-run**
  (it logs, never sends). This is intentional, not a misconfiguration.
- The **operator agent (CC)** is the bridge: it pulls approved clips from
  ClipForge (via the `clipforge` MCP tools), then calls **Hermes' own MCP tools**
  to post/DM on the Android farm — after your Telegram approval.

```
ClipForge (produce + HITL gate)  ──MCP──▶  CC operator  ──MCP──▶  Hermes agent ──▶ phones
        approve_clip / review_queue                              (posting + DMs)
```

So the two HITL gates still hold: nothing reaches a real account until you
approve in Telegram, and the actual send is Hermes' job.

## Wire the operator agent (CC)

```bash
# add the ClipForge MCP server to CC (user scope)
claude mcp add clipforge -s user \
  -e CLIPFORGE_API_URL=http://127.0.0.1:3000 \
  -- node /opt/clipforge/src/mcp/server.mjs

# load the operator skill
mkdir -p /root/.claude/skills
cp -r /opt/clipforge/skills/clipforge-operator /root/.claude/skills/
```

Then from Telegram: ask CC to run `brand` or `review_queue` to confirm the wiring.
